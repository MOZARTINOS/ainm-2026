## 1. Tripletex SupplierInvoice вЂ” Entity Creation and Ledger Mapping

The creation of a supplier invoice ("leverandГёrfaktura") in the Tripletex system is often a source of confusion for developers, primarily because the API abstracts the concept of a "Supplier Invoice" differently from the graphical user interface (GUI). 

### 1.1 Direct Answer and API Endpoint Details
While there is a `/supplierInvoice` endpoint in the Tripletex v2 API, research indicates that it is predominantly used for reading, attesting (approval workflows), and updating the payment status of invoices in the "voucher reception" (mottak) [cite: 1, 2]. **You do not POST to `/supplierInvoice` to create a new accounting entry for an incoming invoice.** 

Instead, a supplier invoice is auto-generated by the system when you successfully create a voucher via the `POST /ledger/voucher` endpoint [cite: 1, 8]. The relationship is fundamentally causal: the `/ledger/voucher` endpoint represents the underlying double-entry accounting journal. When a voucher includes a posting against a supplier ledger account (typically account 2400 "LeverandГёrgjeld"), the system implicitly recognizes this transaction as a supplier invoice and populates the `/supplierInvoice` endpoints accordingly.

### 1.2 Required Fields and Account Structure
Norwegian accounting relies on the *Norsk Standard Kontoplan*. When registering a standard incoming supplier invoice with 25% VAT, the correct accounting structure requires a balanced voucher (sum of postings equals zero):

1.  **Credit (LeverandГёrgjeld - Supplier Debt):** Account 2400. Because this account has a `ledgerType` of `SUPPLIER`, the posting *must* include a reference to the `supplierId` [cite: 8]. The amount is entered as a negative number (Credit).
2.  **Debit (Expense Account):** e.g., Account 6500 (VerktГёy/Maskiner) or 6800 (Kontorrekvisita). The amount is a positive number (Debit).
3.  **VAT Handling:** You do not manually post to account 2710 (InngГҐende mva) unless you are doing a direct, non-standard manual entry (which does not properly hit the VAT return) [cite: 4]. Instead, you attach a `vatType` ID to the expense posting. For 25% inbound VAT, the correct `vatType` ID is usually `1` [cite: 9]. The Tripletex engine will automatically split the tax and post the 25% to account 2710 on the backend.

### 1.3 Code Example: POST /ledger/voucher
To successfully score 8/8 on a supplier invoice task, your JSON payload for `POST /ledger/voucher` should structurally resemble the following:

```json
{
  "date": "2026-03-20",
  "description": "InnkjГёp av kontorrekvisita",
  "voucherType": {
    "id": 1 // Often 1 for standard vouchers or 3 for incoming invoices depending on setup
  },
  "postings": [
    {
      "account": {
        "id": 2400 // The actual database ID of account 2400 in the specific tenant
      },
      "supplier": {
        "id": 12345 // Must be created beforehand
      },
      "amountGross": -12500.00,
      "amountGrossCurrency": -12500.00
    },
    {
      "account": {
        "id": 6800 // The actual database ID of the expense account
      },
      "vatType": {
        "id": 1 // Represents 25% Inbound VAT
      },
      "amountGross": 12500.00,
      "amountGrossCurrency": 12500.00
    }
  ]
}
```

### 1.4 Common Pitfalls
*   **"LeverandГёr mangler" (Supplier missing) Error:** This 422 Bad Request error occurs if you post to account 2400 but forget to include the `{"supplier": {"id": xxx}}` object in the posting [cite: 8].
*   **Direct VAT Posting:** Directly debiting account 2710 does not populate the statutory VAT return (MVA-meldingen). The correct protocol is to debit the gross amount to the expense account and apply `vatType: 1` [cite: 4].
*   **Internal IDs vs. Account Numbers:** In the API payload, `"id": 2400` typically refers to the *database ID* of the account, not the literal account string "2400". You must perform a `GET /ledger/account?number=2400` to retrieve the primary key ID for that specific sandbox tenant [cite: 10].

## 2. Tripletex Payroll / Salary (LГёnn) Integration

Processing payroll autonomously requires high-precision accounting due to statutory reporting obligations (A-melding) to the Norwegian government. 

### 2.1 Direct Answer and API Endpoint Details
While rudimentary salary vouchers can be brute-forced via manual `POST /ledger/voucher` calls, Tripletex features a dedicated API module for payroll. Research confirms the existence of endpoints such as `POST /salary/transaction`, `GET /salary/type`, and `/salary/payslip` [cite: 7, 11]. However, if the NM i AI task specifically tests the agent's ability to construct a *payroll voucher* (as opposed to running a full automated payroll batch), the agent must construct a manual ledger voucher capturing all payroll mechanics.

### 2.2 Required Fields and Account Structure
If executing via `POST /ledger/voucher`, the employee **must** be created before the voucher. Ledger accounts linked to the `EMPLOYEE` ledgerType mandate an `employee` ID reference [cite: 8].

The standard Norwegian payroll accounting flow requires balancing the following:
*   **Account 5000 (LГёnn til ansatte):** Debit (Gross Salary). This represents the total expense for the company.
*   **Account 2920 (Forskuddstrekk / Skattetrekk):** Credit (Employee Tax Deduction).
*   **Account 1920 (Bankinnskudd):** Credit (Net payout to the employee).
*   **Account 5400 (Arbeidsgiveravgift):** Debit (Employer's social security contribution, typically 14.1% in standard zones).
*   **Account 2770 (Skyldig arbeidsgiveravgift):** Credit (The accrued liability to the government for the employer tax).

*Note: The arbeidsgiveravgift (14.1%) is an additional cost calculated on top of the gross salary (Account 5000) and does not affect the employee's net payout.*

### 2.3 Code Example: Payroll Voucher Payload
```json
{
  "date": "2026-03-25",
  "description": "LГёnn Mars 2026",
  "postings": [
    {
      "account": { "id": 5000 }, // Gross Salary
      "employee": { "id": 98765 }, // Required for EMPLOYEE ledger type
      "amountGross": 50000.00
    },
    {
      "account": { "id": 2920 }, // Tax Withheld
      "employee": { "id": 98765 },
      "amountGross": -15000.00
    },
    {
      "account": { "id": 1920 }, // Net Pay to Bank
      "amountGross": -35000.00
    },
    {
      "account": { "id": 5400 }, // Employer Tax Expense (14.1% of 50000)
      "amountGross": 7050.00
    },
    {
      "account": { "id": 2770 }, // Employer Tax Payable
      "amountGross": -7050.00
    }
  ]
}
```

### 2.4 Common Pitfalls
*   **Missing Employee Reference:** Just like the supplier invoice, omitting the employee ID on account 5000 or 2920 will trigger a 422 "Ansatt mangler" (Employee missing) validation error [cite: 8].
*   **Mathematical Imbalance:** The sum of all postings in the voucher must perfectly equal zero. Miscalculating the 14.1% `arbeidsgiveravgift` or the net pay will cause the API to reject the voucher.
*   **Module Activation:** In a fresh sandbox, the salary module or advanced accounting modules may need to be activated before certain system accounts (like 2920 or 2770) are fully accessible. 

## 3. Tripletex Travel Expense Scoring

Travel expenses (Reiseregning) represent a composite entity in Tripletex, heavily scrutinized by the scoring algorithm. An aggregate score of 4.5/8 despite "successful" API calls indicates that while the database accepts the entities, statutory or logical fields required for full compliance are missing or malformed.

### 3.1 Direct Answer and API Endpoint Details
The creation of a travel expense report involves a hierarchical API flow:
1.  `POST /travelExpense` (Creates the parent container)
2.  `POST /travelExpense/cost` (Adds individual receipts/costs) [cite: 12, 13]
3.  `POST /travelExpense/perDiemCompensation` (Adds daily allowances based on state rates) [cite: 6]

### 3.2 Critical Fields and Relationships
**Parent `travelExpense`:** Requires `employeeId`, `title`, `departureDate`, `returnDate`, and `state` (usually set to `DELIVERED` or `APPROVED` depending on the prompt).
**`travelExpense/cost`:** The scoring algorithm checks `amountNOKInclVAT`, `costCategory`, and `paymentType` [cite: 12]. The distinction between `costCategory` and `paymentType` is critical [cite: 12, 13]. `paymentType` defines if the employee paid out-of-pocket (needs reimbursement) or with a corporate card.
**`travelExpense/perDiemCompensation`:** Critical fields include `location` (domestic vs international), `overnightAccommodation` (hotel vs private, which dictates the rate), and `count` (number of days). Tripletex introduced a `/travelExpense/perDiemCompensation/autoSuggest` endpoint to dynamically calculate these complex rates based on travel details [cite: 6].

Cost Category IDs and Payment Type IDs are generally standardized across Tripletex environments but should ideally be queried dynamically via `GET /travelExpense/costCategory` and `GET /travelExpense/paymentType` to avoid hardcoding errors [cite: 12].

### 3.3 Code Example: Cost and Per Diem
```json
// POST /travelExpense/cost
{
  "travelExpense": { "id": 1001 },
  "costCategory": { "id": 2 }, // e.g., Taxi/Transport
  "paymentType": { "id": 1 }, // e.g., Paid by employee
  "date": "2026-03-10",
  "title": "Taxi til flyplass",
  "amountNOKInclVAT": 450.00,
  "rate": 1.0
}

// POST /travelExpense/perDiemCompensation
{
  "travelExpense": { "id": 1001 },
  "location": "Oslo",
  "overnightAccommodation": "HOTEL",
  "count": 2,
  "rateType": "DOMESTIC" 
}
```

### 3.4 Common Pitfalls
*   **amountNOKInclVAT vs amountCurrencyInclVAT:** Historically, developers used deprecated fields for costs. The required field for local currency is `amountNOKInclVAT` [cite: 12].
*   **Employee Entitlements:** The employee submitting the travel expense must have the correct user profile/entitlements. If the agent creates a generic employee without activating the "Travel Expense" module access, the background validation may penalize the score [cite: 10, 11].

## 4. Tripletex Invoice Multi-Line with Products

Scoring 3/8 on a multi-line invoice suggests that while the invoice is being generated and sent, the line-item granularity, VAT mappings, or product instantiation is flawed.

### 4.1 Direct Answer and API Endpoint Details
To achieve a perfect score, products should be created properly via `POST /product`, added to an order via `POST /order/orderline`, and finally invoiced via `PUT /order/{id}/:invoice` [cite: 1, 14, 15].

### 4.2 Required Fields and VAT Mappings
**Product Creation:** The critical fields for `POST /product` are `name`, `number` (SKU), `priceExcludingVatCurrency`, and `vatType`. The `number` field is often explicitly checked by the scoring engine if the prompt specifies a product number like "8944" [cite: 3, 16].
**VAT Types (Outbound/Sales):**
*   **25% Standard Rate:** vatType ID `3`
*   **15% Medium Rate (Food/Beverage):** vatType ID `31`
*   **11.11% Raw Fish (Special):** vatType ID `32` (or similar depending on update)
*   **0% Exempt/Export:** vatType ID `5`
*   *(Note: IDs 1, 11, 12 are for INBOUND/Purchasing VAT. Using inbound VAT codes on an outbound sales order will cause accounting errors and score reductions)* [cite: 9].

### 4.3 Code Example: Orderlines and Invoicing
```json
// POST /order/orderline
{
  "order": { "id": 5050 },
  "product": { "id": 3030 }, // Reference the created product ID
  "count": 5,
  "unitPriceExcludingVatCurrency": 100.00,
  "vatType": { "id": 3 } // 25% Outbound VAT
}
```

To finalize the invoice:
```json
// PUT /order/5050/:invoice?invoiceDate=2026-03-20&sendToCustomer=true&sendType=EMAIL
// (Note: PUT actions use query parameters for actions in Tripletex, rather than JSON bodies for the execution commands)
```
Tripletex allows overriding the send type directly in the `:invoice` endpoint using `sendType` and `overrideEmailAddress` [cite: 15].

### 4.4 Common Pitfalls
*   **Inline vs Referenced Products:** Creating products inline inside the orderLine array often skips product database registration, scoring 0 points for the product creation check. Always use `{ "product": { "id": X } }` to reference pre-created products.
*   **VAT Inclusivity Conflicts:** A major pitfall is providing `unitPriceExcludingVatCurrency` on an order line when the parent order is configured as `isPrioritizeAmountsIncludingVat = true`. This mismatch results in a 422 error [cite: 17].
*   **Order Date vs Delivery Date:** Delivery date (`deliveryDate`) is statutorily required on Norwegian invoices to determine the correct VAT period. If the prompt specifies a delivery date different from the invoice date, failing to set it will lose points [cite: 14, 17].

## 5. Tripletex Project Invoice with Hourly Rates

Scoring 0/8 despite executing the full chain (project -> activity -> timesheet -> order -> invoice) points to a breakdown in relational mapping or pricing hierarchy. 

### 5.1 Direct Answer and API Endpoint Details
The correct flow is indeed: `Customer -> Employee -> Project -> Activity -> ProjectActivity -> TimesheetEntry -> Order -> Invoice`. 
However, the linkage of **Hourly Rates** is the missing link. Rates must be configured via `PUT /project/hourlyRates/projectSpecificRates` or `POST /project/hourlyRates/projectSpecificRates/list` [cite: 6, 18].

### 5.2 Required Fields and Linkages
*   **Project Manager:** For strict compliance and scoring, the project should have `projectManager` set. Often the prompt implies who the manager is [cite: 19].
*   **Project Activity (`/project/projectActivity`):** An activity (e.g., "Consulting") must be explicitly linked to the project [cite: 18].
*   **Timesheet Entry (`/timesheet/entry`):** Critical fields are `date`, `employee`, `project`, `activity`, `hours`, and `chargeableHours`. If `chargeableHours` is 0 or null, the hours cannot be invoiced [cite: 11].
*   **Invoicing:** You do not manually create the order lines for timesheets. You use the project invoicing workflows or generate the order from the billable project hours.

### 5.3 Code Example: Hourly Rates and Timesheet
```json
// POST /project/hourlyRates/projectSpecificRates
{
  "project": { "id": 4040 },
  "activity": { "id": 101 },
  "hourlyRate": 1500.00,
  "startDate": "2026-01-01"
}

// POST /timesheet/entry
{
  "employee": { "id": 999 },
  "project": { "id": 4040 },
  "activity": { "id": 101 },
  "date": "2026-03-20",
  "hours": 5.0,
  "chargeableHours": 5.0
}
```

### 5.4 Common Pitfalls
*   **Timesheet Approval:** Depending on sandbox settings, timesheets may need to be approved (`PUT /timesheet/entry/{id}/:approve`) before they can be invoiced.
*   **Hourly Rate Override:** If hourly rates are not explicitly defined at the project level, Tripletex falls back to employee or activity base rates, resulting in the wrong monetary value on the final invoice, yielding a 0 score for financial correctness.

## 6. Efficiency Optimization Strategies

The NM i AI scoring formula is: `Score = Correctness Г— Tier Multiplier Г— (1 + Efficiency Bonus)` [cite: 3]. The efficiency bonus is highly sensitive and separates top competitors.

### 6.1 Exact Formula Mechanics
While the exact mathematical weights remain proprietary to the competition platform, the documentation outlines the mechanics clearly:
*   The bonus applies **only** if the Correctness score is a perfect 1.0 [cite: 3].
*   **Call Efficiency:** Measured by comparing your agent's total API request count (GET + POST + PUT + DELETE) against the "best known solution" for that task. If the community best is 4 calls, and your agent takes 12, your bonus approaches zero [cite: 3].
*   **Error Cleanliness:** Every 4xx error (e.g., 400 Bad Request, 404 Not Found, 422 Validation Error) heavily penalizes the bonus. Retrying a 422 successfully achieves correctness, but destroys the efficiency bonus. "An agent that gets it right without trial-and-error is rewarded" [cite: 3].

### 6.2 Cache Warming and Minimum Calls
*   **Cache Warming:** Because every API call routed through the proxy is logged and counted, "cache warming" (e.g., executing a `GET /department` or `GET /ledger/account` blindly at the start of every task) will bloat your call count and reduce your score.
*   **Minimum API Call Count:** There is no hardcoded minimum. It is determined dynamically by the leaderboard. To achieve the best score, your agent must deduce internal IDs without redundant searching, or utilize batch endpoints where possible [cite: 3].

### 6.3 Algorithmic Optimization Strategies
1.  **Zero-Shot JSON Generation:** The LLM (Gemini 2.5 Flash) should ideally generate the exact, perfect JSON payloads for all necessary endpoints in a single inference step, rather than operating in a ReAct loop that makes an API call, encounters an error, and corrects itself.
2.  **Deterministic ID Resolution:** Instead of querying `GET /ledger/account?number=2400`, provide the LLM with a pre-mapped dictionary of standard Tripletex account numbers to their standard database IDs (if they are static across sandboxes). If IDs are dynamic per sandbox, use a single, highly optimized `GET /ledger/account` query with comma-separated values or broad filters to fetch all needed accounts in **one** call.
3.  **Batch Endpoints:** Use endpoints like `POST /employee/list` or `POST /customer/list` to create multiple entities in a single HTTP request [cite: 7].

## 7. Tripletex Custom Dimensions (Fri Regnskapsdimensjon)

Scoring 0/13 on `dimension_voucher` implies a complete failure to register or link the custom dimension.

### 7.1 Direct Answer and API Endpoint Details
Tripletex introduced specific endpoints for user-defined dimensions: `/accountingDimensionName` and `/accountingDimensionValue` [cite: 6, 11].
*   `POST /accountingDimensionName` creates the structural category (e.g., "Region", "Campaign"). You are limited to a maximum of 3 dimensions, which will automatically be assigned indexes 1, 2, and 3 [cite: 6].
*   `POST /accountingDimensionValue` creates the actual selectable tags (e.g., "North", "South") under that dimension name [cite: 6].

### 7.2 Linkage to Vouchers
Once created, these dimensions are linked to ledger postings via the fields `freeAccountingDimension1`, `freeAccountingDimension2`, and `freeAccountingDimension3` within the posting DTO [cite: 6].

### 7.3 Code Example: Dimensions
```json
// POST /accountingDimensionName
{
  "name": "Kampanje",
  "description": "MarkedsfГёringskampanjer"
}
// Assume this is assigned index 1 by the API.

// POST /accountingDimensionValue
{
  "accountingDimensionName": { "id": 10 }, // ID from previous call
  "name": "Sommer2026",
  "number": "S26"
}

// Inside POST /ledger/voucher posting array:
{
  "account": { "id": 3000 },
  "amountGross": -10000,
  "freeAccountingDimension1": { "id": 50 } // ID of the created Dimension Value
}
```

### 7.4 Common Pitfalls
*   **Package Limitations:** Free accounting dimensions are *only* available to customers with the "Pro" package or above. They are not available on "Smart" or "Basis" [cite: 6]. If the NM i AI sandbox defaults to a lower tier, the agent may need to programmatically upgrade the subscription or activate the module via `PUT /company/modules` or `/subscription` before making the call [cite: 11, 20].
*   **Permissions:** Executing these endpoints requires the specific role "Regnskapsinnstillinger, kontoplan og historisk balanse" [cite: 6]. Ensure the employee token used has Administrative rights [cite: 3, 10].

## 8. General Tripletex Best Practices for AI Agents

To elevate the score from 23.1 to the theoretical maximum (up to 180), the agent architecture must adhere to strict entity creation ordering and leverage deeper Tripletex domain knowledge.

### 8.1 Recommended Order of Entity Creation
Because Tripletex enforces strict foreign-key constraints in its SQL backend, the dependency graph must be resolved from the bottom up:
1.  **System/Settings:** Dimensions, Departments, VAT settings.
2.  **Core Entities:** Employees, Customers, Suppliers.
3.  **Inventory/Projects:** Products, Projects, Activities.
4.  **Transactional Entities:** Orders, OrderLines, Timesheets, Travel Expenses.
5.  **Accounting:** Invoices, Vouchers, Payments.

### 8.2 Bank Account Setup
Should bank account (1920) be set up before creating invoices? While you can post to account 1920 without configuring a full bank integration, setting up the actual bank account entity in Tripletex (via `/bank`) is generally required if the task involves *bank reconciliation* (`/bank/reconciliation`) or executing outward payments [cite: 7]. For simple invoice creation, it is not strictly required.

### 8.3 `isCustomer` vs `isSupplier` Flags
The boolean flags `isCustomer` and `isSupplier` on the contact/company endpoints dictate visibility across the system [cite: 12]. A company created with `isCustomer=true` will be assigned a customer ledger number (e.g., 10000) and can be referenced in order creation. If you try to create a Supplier Invoice against an entity where `isSupplier=false`, the API will return a 422 validation error [cite: 8]. Always set both to `true` if the prompt implies a dual relationship, or strictly set the correct flag to save API validation headaches.

### 8.4 Relationship between Department and Employee
A `department` (Avdeling) is a foundational cost center in Tripletex. When creating an `employee`, assigning them to a department is best practice and often mandatory if the company has enabled departmental accounting. Creating the department via `POST /department` [cite: 7] and referencing its ID in the `POST /employee` payload prevents downstream errors when processing payroll or travel expenses, which often inherit the employee's default department for cost allocation.

### 8.5 Eliminating 401/403 Errors
Ensure that the authentication header is formulated correctly. As noted in the documentation, Basic Auth should be used where the username is `0` (denoting the company of the employee) and the password is the `sessionToken` [cite: 16, 21]. Bypassing authorization failures is the first step toward perfect Error Cleanliness.

### Conclusion for AI Architecture Adjustment
To drastically improve the current score of 23.1, modify the Gemini 2.5 Flash prompt to output a **Directed Acyclic Graph (DAG)** of API calls rather than a single endpoint action. The LLM must understand that "Create a travel expense" means generating an array of sequential requests (`/travelExpense` $\rightarrow$ `/travelExpense/cost` $\rightarrow$ `/travelExpense/perDiemCompensation`). Implement a strict validation layer in the JavaScript execution engine that catches missing foreign-key IDs (like `supplierId` on account 2400) *before* the HTTP request is dispatched, preserving the critical Efficiency Bonus by maintaining an error count of zero.
